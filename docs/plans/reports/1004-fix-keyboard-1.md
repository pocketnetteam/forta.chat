## 1. Переработка управления высотой клавиатуры

**Файлы:** `keyboard-height.ts`, `App.vue`, `main.css`, `MainActivity.kt`

### Проблема
Единственная функция `computeKeyboardHeight` не учитывала порядок приоритета источников событий клавиатуры. При одновременном получении native-событий (WindowInsetsCompat) и web-событий (visualViewport / VirtualKeyboard API) возникал jitter — двойные обновления CSS-переменной `--keyboardheight`, из-за чего контент дёргался.

### Решение
- Введён класс **`KeyboardHeightManager`** с приоритетной моделью источников:
  - **native** (WindowInsetsCompat из MainActivity.kt) — высший приоритет
  - **VirtualKeyboard API** (экспериментальный браузерный API) — средний
  - **visualViewport** — базовый fallback
- После native-события web-обновления **игнорируются на 120ms** (native lock), что устраняет jitter от перекрывающихся источников.
- `scrollIntoView` при фокусе теперь **условный**: вызывается только если элемент действительно перекрыт клавиатурой (`isOccludedByKeyboard`), а не всегда. Задержка увеличена до 350ms для ожидания анимации открытия клавиатуры.
- В CSS `safe-bottom` используется вложенный `max()` для совместимости со старыми WebView (3-аргументный `max()` не всегда поддерживается).
- В `MainActivity.kt` добавлена новая CSS-переменная `--native-safe-area-inset-bottom` для корректного учёта safe area на устройствах с вырезами.
- Добавлена debug-телеметрия (`window.__FORTA_KB_DEBUG = true`) для диагностики проблем с клавиатурой на устройствах.

## 2. Throttling `getnodeinfo` RPC (10 минут)

**Файлы:** `app-initializer.ts`, `App.vue`

### Проблема
Каждый вызов `syncNodeTime()` и `getBlockHeight()` делал отдельный RPC `getnodeinfo` к ноде. При частых вызовах (каждые 60 секунд для block height + при каждом resume/focus) это создавало избыточную нагрузку на сеть.

### Решение
- Введён **статический кэш** `_lastNodeInfo` с TTL 10 минут, общий для всех экземпляров `AppInitializer`.
- Параллельные вызовы **разделяют один in-flight промис** (`_nodeInfoPromise`).
- При ошибке RPC возвращается **stale-кэш вместо throw** — устаревшие данные лучше, чем каскадный сбой (failed getnodeinfo → syncNodeTime fails → actions.prepare() not called → proxy stale → emptykey).
- Интервал обновления block height увеличен с 60s до **10 минут**.
- Добавлена синхронизация времени ноды при `visibilitychange` (tab focus) и `appStateChange` (Capacitor resume).

## 3. RPC Batcher — батчинг RPC-запросов

**Файлы:** `rpc-batcher.ts` (новый), `rpc-batcher.test.ts` (новый), `app-initializer.ts`

### Проблема
`loadPostScores` делал отдельный RPC `getpostscores` для каждого поста при прокрутке ленты канала.

### Решение
- Создан универсальный **`RpcBatcher`** (DataLoader-паттерн): собирает `load(key)` вызовы в окне 50ms, отправляет один батч-запрос, раздаёт результаты обратно по ключам.
- `loadPostScores` теперь использует `RpcBatcher` — все посты в видимой области батчатся в один `getpostscores`.
- Покрыт тестами: батчинг, роутинг по ключам, дедупликация, обработка ошибок, кастомный delay.

## 4. Кэш криптографических профилей (getUsersInfo)

**Файлы:** `stores.ts` (auth)

### Проблема
При каждом decrypt/encrypt для комнаты вызывался `getUsersInfo`, который делал сетевые запросы `loadUsersInfo` + `loadUsersInfoRaw` для всех участников. Для комнат с повторяющимися участниками это вызывало десятки дублирующих RPC.

### Решение
- Введён **`_cryptoProfileCache`** — in-memory кэш с 7-дневным TTL для результатов `getUsersInfo`.
- Адреса делятся на cached / uncached; сетевые запросы отправляются **только для uncached**.
- **Защита от пустых ключей:** кэшируются только записи с `keys.length >= 12`. Если новый fetch вернул пустые ключи — используется stale-кэш (пустые ключи = emptykey ошибка при шифровании).
- Собственный адрес во время регистрации **никогда не кэшируется** (профиль может измениться в любой момент).
- В catch-блоке при полном сбое также предпочитается stale-кэш.
- Логика построения `userInfoEntry` вынесена в отдельную функцию `_buildUserInfoEntry`.

## 5. Оптимизация User Store

**Файлы:** `user-store.ts`, `user-store.test.ts` (новый)

### Проблема
- Собственный адрес пользователя мешался в один батч с чужими адресами при загрузке профилей, что могло вызывать проблемы с pSDK (например, при регистрации).
- TTL кэша профилей был 6 часов — слишком частые обновления.

### Решение
- **Собственный адрес всегда батчится отдельно** от остальных адресов.
- Во время **регистрации** собственный адрес **всегда перезагружается** из сети (игнорируя кэш), а `cachedAt` не ставится, чтобы следующий вызов тоже сходил в сеть.
- TTL профилей увеличен с **6 часов до 7 дней** (stale-while-revalidate — UI никогда не блокируется).
- Вынесена общая функция `_fetchAndStore` для дедупликации кода загрузки/сохранения профилей.
- Добавлены юнит-тесты: разделение батчей, cachedAt при регистрации, TTL-aware пропуск, принудительная перезагрузка.

## 6. Кэш профиля при логине (fetchUserInfo)

**Файлы:** `stores.ts` (auth)

### Проблема
При каждом открытии приложения `fetchUserInfo` делал полный сетевой запрос для загрузки профиля пользователя, даже если профиль уже был закэширован в localStorage.

### Решение
- **Fast path:** если в `userStore` есть свежий кэшированный профиль (< 7 дней) и регистрация не в процессе — профиль берётся из кэша без сетевого вызова.
- pSDK всё равно инициализируется **в фоне** (`initializeAndFetchUserData().catch(() => {})`), чтобы `getUserData()` работал для последующих вызовов.

## 7. Кэш верификации ключей (verifyAndRepublishKeys)

**Файлы:** `stores.ts` (auth)

### Проблема
Верификация 12 опубликованных ключей шифрования выполнялась при каждом логине, делая RPC к блокчейну.

### Решение
- Результат верификации **кэшируется в localStorage** с 7-дневным TTL.
- При следующем логине RPC пропускается, если верификация была успешной менее 7 дней назад.
- Кэш записывается как при успешной проверке через SDK-кэш, так и при проверке через blockchain RPC.

## 8. Session guards в chat-store (предотвращение повторных scrollback)

**Файлы:** `chat-store.ts`, `chat-store-session-guards.test.ts` (новый), `chat-store-no-scrollback.test.ts` (новый), `MessageList.vue`

### Проблема
При возвращении в уже загруженную комнату `loadRoomMessages` снова делал scrollback (/messages API), даже если таймлайн уже был полностью загружен в этой сессии. Аналогично `prefetchNextBatch` повторно запрашивал старые сообщения.

### Решение
- Введены **session-guard множества:**
  - `_roomsWithLoadedTimeline` — комнаты, где scrollback уже выполнен; при повторном входе scrollback-цикл пропускается.
  - `_roomsPrefetched` — комнаты, где prefetch уже выполнен.
- Guards **сбрасываются** при `RECONNECTING`, `STOPPED` и `PREPARED` (initial/full sync), чтобы после переподключения данные обновились.
- `fetchRoomPreview` (для списка контактов) теперь **не делает network scrollback** — только Dexie-кэш. Полный scrollback выполняется только при фактическом открытии комнаты.
- `MessageList.vue` проверяет `isRoomTimelineLoaded()` перед вызовом `loadMessages()`.

## 9. Дедупликация read receipts

**Файлы:** `chat-store.ts`, `chat-store-receipts.test.ts`

### Проблема
При повторном открытии уже прочитанной комнаты отправлялись дублирующие `/read_markers` запросы на сервер — watermark не изменился, но рецепт всё равно отправлялся.

### Решение
- Введён `_lastCommittedReceiptTs` — Map, хранящий timestamp последнего отправленного рецепта для каждой комнаты.
- `commitReadWatermark` **пропускает отправку**, если timestamp не превышает последний отправленный.
- `flushPendingReadWatermarks` также проверяет `_lastCommittedReceiptTs` и удаляет уже неактуальные pending-записи.
- При входе в комнату (`setActiveRoom`) вызывается `initReceiptWatermark`, который **сидит из Dexie** (`lastReadInboundTs`) для предотвращения повторной отправки при первом визите.

## 10. Исправления в Channel Store

**Файлы:** `channel-store.ts`, `ChannelList.vue`

### Изменения
- Глобальный `blockHeight` заменён на **per-channel `sessionHeight`** (Map) — каждый канал отслеживает свою высоту блока из ответа `getProfileFeed`.
- `getProfileFeed` теперь возвращает `{ posts, height }` вместо просто массива постов.
- Удалён **prefetch постов при скролле списка каналов** (`prefetchVisiblePosts`) — посты загружаются только при выборе канала (`setActiveChannel` всегда вызывает `fetchPosts(address, true)`).
- `fetchChannels` передаёт `height: 0` вместо кэшированного blockHeight (серверная пагинация управляет этим сама).

## 11. Исправление дублирования отправки сообщений

**Файлы:** `MessageInput.vue`

### Проблема
При быстром двойном нажатии кнопки отправки сообщение могло отправиться дважды, т.к. forward-состояние очищалось асинхронно.

### Решение
- Текст и forward-состояние **очищаются синхронно** в начале `handleSend()`, до любых async-вызовов. Повторное нажатие попадает на guard `text === ""` и выходит.
- Forward-данные захватываются в локальную переменную перед очисткой store.
- При неудачной отправке (`inserted === false`) текст и forward-состояние **восстанавливаются**.
- После отправки textarea получает фокус (`textareaRef.value?.focus()`).

## 12. Загрузка профилей для системных сообщений

**Файлы:** `MessageList.vue`

### Проблема
Системные сообщения (invite, kick, ban и т.д.) ссылаются на пользователей, которые могут не быть текущими участниками комнаты. `loadProfilesForRoomIds` загружает профили только для текущих участников, поэтому имена в системных сообщениях отображались как обрезанные хеши.

### Решение
- Добавлен watcher на `activeMessages`, который собирает `senderAddr`/`targetAddr` из `systemMeta` и загружает недостающие профили через `userStore.enqueueProfiles()`.
- Для отображения используется `isUnresolvedName()` — если имя является обрезанным адресом/хешем, показывается `t("common.unknownUser")`.

## 13. Прочие исправления

### Сортировка комнат при очистке истории (`room-repository.ts`)
- При `clearHistory` больше **не обнуляется `lastMessageTimestamp`** — это сохраняет позицию сортировки комнаты. Ранее обнуление вызывало fallthrough на `updatedAt`, который дрейфует вверх при синхронизации, и комната "прыгала" вверх в списке.

### Timestamp для комнат без сообщений (`chat-store.ts`)
- Для **всех** комнат (не только invite) при отсутствии timeline-событий используется timestamp события `m.room.create` вместо 0.

### `loadUserData` и `loadUsersBatch` переведены на light mode
- Оба метода в `app-initializer.ts` теперь вызывают `psdk.userInfo.load(addresses, true)` — light mode использует более крупные батчи (70 vs 10) и queue-based загрузку.

### `.gitignore`
- Добавлен `*.har` (HAR-файлы сетевых трейсов).

### Диагностика в `matrix-crypto.ts`
- Добавлены error-логи при пустом `preparedUsers` в `encryptEvent` и при отсутствии target user в `_encrypt` — помогает диагностировать emptykey-ошибки.
